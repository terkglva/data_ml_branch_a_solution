# Тестовое задание: классификация и граф контрагентов

Выбранная ветка Части 2: **Ветка A — Data / ML**

## Как запустить

Положите входные файлы в папку:

```text
data/
├── transactions.csv
├── categories.json
└── gold_set.csv
```

Эти файлы не входят в репозиторий и должны быть добавлены вручную.

Создание окружения и установка зависимостей:

```bash
python -m venv .venv
.venv\Scripts\activate && pip install -r requirements.txt
```

Запуск:

```bash
jupyter notebook
```

Откройте:

```text
notebooks/Data_ML_solution.ipynb
```

Для self-check выполните:

```text
Kernel → Restart & Run All
```

### Базовая часть

- EDA: размер датасета, период, уникальные контрагенты
- Топ-20 контрагентов по входящему и исходящему обороту
- Распределение сумм в log-scale и помесячная динамика
- Поиск аномалий:
  - невалидные БИН/ИИН
  - дубли
  - пропуски описаний
  - суммы <= 0
  - экстремальные значения
- Очистка данных:
  - валидация БИН/ИИН (12 цифр + контрольная сумма РК)
  - нормализация дат
  - удаление дублей
- Таблица "было/стало"
- SQL-анализ:
  - топ-10 пар контрагентов
  - rolling sum через window functions
  - контрагенты с >70% входящих платежей от одного источника

### Ветка A — Data / ML

- Классификация `description` по категориям из `categories.json`
- Keyword/Regex baseline
- TF-IDF + Logistic Regression
- Precision / Recall / F1 по каждому классу
- Confusion Matrix
- Анализ путаниц между категориями
- Confidence score
- Threshold для ручной проверки:

```text
confidence_ml < 0.45
```
## Краткие выводы
- Исходный файл содержит **80 800** строк, после удаления дублей осталось **80 000** строк.
- Период данных: **2024-01-01 — 2026-04-30**.
- Уникальных контрагентов после очистки: **7 215**.
- Доля строк, где оба идентификатора прошли контрольную сумму: **97.13%**.
- Удалено дублей по бизнес-ключу: **800**.
- Пропуски в `description` после очистки: **1.04%**.
- Keyword baseline: **macro-F1 = 0.915**.
- TF-IDF + Logistic Regression: **macro-F1 = 0.883**.
- При `threshold = 0.45` на ручную проверку уходит **21.5%** примеров.
- Основные ошибки классификации возникают между близкими сервисными категориями:
  - Юридические услуги ↔ Консалтинговые услуги
  - Бухгалтерские и аудиторские услуги ↔ Коммунальные услуги
  - Спецодежда и СИЗ ↔ Медицинские товары и препараты
- Для улучшения качества модели можно расширить словарь исключений, добавить реальные размеченные описания и отдельно обработать слишком общие слова вроде «услуги», «консультация», «поставка».
Почему ML ниже baseline
Доступный `gold_set.csv` используется только для оценки качества. Чтобы не обучаться и тестироваться на одном и том же наборе, ML-модель обучается на слабой синтетической разметке, сгенерированной из `categories.json`. Поэтому keyword baseline на таком словарном датасете может быть сильнее. В реальном решении качество ML должно улучшиться после добавления размеченных реальных транзакций.
